# Telemetry Event Contract v1

**Status:** Draft  
**Schema version:** `1`  
**Document:** `telemetry-event-v1`

---

## Overview

All observability data entering Obsyl must conform to this telemetry event contract. The contract defines a single, canonical envelope for logs, metrics, and traces — regardless of how or where the data originates.

This document specifies **data formats only**. It does not prescribe storage engines, serialization libraries, or service implementation details.

The contract is designed to be consumed consistently across the platform:

| Consumer | Role |
| -------- | ---- |
| **HTTP ingestion APIs** | Validate and accept telemetry from external clients and SDKs |
| **Kafka event streams** | Publish and consume events between ingestion, processing, and query services |
| **Storage systems** | Index, partition, and retain telemetry by event type and metadata |
| **Internal microservices** | Exchange events without ad-hoc, service-specific payload shapes |

A shared contract prevents schema drift, reduces integration cost, and makes versioning explicit as the platform evolves.

---

## Contract Metadata

Every telemetry event includes the following envelope fields. They apply to all event types.

### `schemaVersion`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes |
| Example | `"1"` |

Identifies which version of this contract the event conforms to. Consumers use this field to select the correct parser, validator, and downstream mapping logic.

Rationale: Observability platforms outlive individual services. Explicit versioning allows the platform to introduce breaking changes in a controlled way without silently corrupting in-flight data.

---

### `eventId`

| Property | Value |
| -------- | ----- |
| Type | `string` (UUID) |
| Required | Yes |
| Example | `"7f3c2a1b-9e4d-4b2a-8c1f-2d6e5a4b3c2d"` |

A globally unique identifier for this event instance. Generated by the producer at emission time.

Rationale: Enables idempotent ingestion, deduplication across retries, and correlation in audit and replay pipelines. Two events with the same `eventId` represent the same logical emission, not two independent observations.

---

### `timestamp`

| Property | Value |
| -------- | ----- |
| Type | `string` (ISO 8601, UTC) |
| Required | Yes |
| Example | `"2026-06-10T14:32:01.123Z"` |

The time the event was observed or emitted, in UTC with millisecond precision.

Rationale: Timestamp is the primary ordering and windowing dimension for observability data. UTC and ISO 8601 avoid ambiguity across services, regions, and storage backends.

---

### `service`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes |
| Example | `"obsyl-ingestion"` |

The logical name of the service that produced the event. This is a stable identifier, not a hostname or container ID.

Rationale: Service identity is fundamental to filtering, ownership, SLO tracking, and incident scoping. A canonical `service` field ensures consistent grouping across logs, metrics, and traces.

---

### `environment`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes |
| Example | `"local"`, `"staging"`, `"prod"` |

The deployment environment in which the producing service was running when the event was emitted.

Rationale: Prevents accidental mixing of production and non-production data in queries, alerts, and dashboards. Environment is a first-class dimension for access control and data retention policies.

---

## Event Types

The `eventType` field discriminates the payload shape. Obsyl v1 supports three event types.

| Value | Purpose |
| ----- | ------- |
| `LOG` | Structured text events describing application behaviour, errors, and diagnostic output |
| `METRIC` | Numeric measurements representing system health, throughput, latency, or business KPIs |
| `TRACE` | Distributed tracing spans representing a unit of work within a request or background job |

### `LOG`

Log events capture discrete, human-readable (or machine-parseable) statements about what happened at a point in time. They are the primary tool for debugging and incident investigation.

### `METRIC`

Metric events capture quantitative measurements over time. They power dashboards, alerting thresholds, and capacity planning. A single metric event represents one observed data point.

### `TRACE`

Trace events capture spans within a distributed trace. They model request flow across service boundaries, enabling latency analysis and dependency mapping.

---

## Log Event Fields

Required when `eventType` is `LOG`.

### `level`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `LOG`) |
| Example | `"INFO"`, `"WARN"`, `"ERROR"` |

The severity of the log entry. Producers should use a consistent, uppercase vocabulary. Recommended values: `DEBUG`, `INFO`, `WARN`, `ERROR`, `FATAL`.

Rationale: Level drives filtering, alert routing, and noise reduction. A normalised severity scale allows platform-wide policies without per-service configuration.

---

### `message`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `LOG`) |
| Example | `"User session created successfully"` |

The primary log content. Should be a concise, actionable description of the event.

Rationale: The message is the human-facing core of a log entry. Structured context belongs in `metadata`; the message should remain readable on its own.

---

## Metric Event Fields

Required when `eventType` is `METRIC`.

### `metricName`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `METRIC`) |
| Example | `"http.request.duration"` |

A dot-delimited identifier for the metric. Names should be stable and descriptive.

Rationale: Consistent naming conventions (`<domain>.<entity>.<measurement>`) prevent metric sprawl and make discovery predictable across services.

---

### `value`

| Property | Value |
| -------- | ----- |
| Type | `number` |
| Required | Yes (for `METRIC`) |
| Example | `142.5` |

The observed numeric measurement at `timestamp`.

Rationale: Metrics are only useful when values are typed consistently. Floating-point is used to accommodate counters, gauges, and histogram bucket values.

---

### `unit`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `METRIC`) |
| Example | `"ms"`, `"bytes"`, `"count"` |

The unit of measurement for `value`.

Rationale: Explicit units prevent misinterpretation during aggregation and cross-service comparison (e.g. confusing milliseconds with seconds).

---

### `tags`

| Property | Value |
| -------- | ----- |
| Type | `object` (string key-value pairs) |
| Required | No |
| Example | `{ "method": "GET", "status": "200" }` |

Dimensions that qualify the metric. Tags enable grouping, filtering, and drill-down in dashboards.

Rationale: Tags are the primary cardinality mechanism for metrics. They should be low-cardinality, stable keys — not unbounded values such as raw user IDs.

---

## Trace Event Fields

Required when `eventType` is `TRACE`.

### `traceId`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `TRACE`) |
| Example | `"a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6"` |

Identifies the entire distributed trace. All spans belonging to the same request share a `traceId`.

Rationale: `traceId` is the join key that links spans across services into a coherent end-to-end view.

---

### `spanId`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `TRACE`) |
| Example | `"span-001"` |

Uniquely identifies this span within its trace.

Rationale: Each unit of work within a trace must be individually addressable for timing analysis and error localisation.

---

### `parentSpanId`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | No |
| Example | `"span-000"` |

The `spanId` of the parent span. Omitted for root spans.

Rationale: Parent-child relationships reconstruct the call tree. This is essential for understanding service dependencies and identifying bottlenecks.

---

### `operationName`

| Property | Value |
| -------- | ----- |
| Type | `string` |
| Required | Yes (for `TRACE`) |
| Example | `"GET /api/v1/logs"` |

A human-readable name for the operation this span represents.

Rationale: Operation names are the primary label in trace visualisations. They should reflect the logical work being performed, not internal implementation details.

---

### `durationMs`

| Property | Value |
| -------- | ----- |
| Type | `number` (integer) |
| Required | Yes (for `TRACE`) |
| Example | `42` |

The elapsed time of the span in milliseconds.

Rationale: Duration is the core signal for latency analysis. Milliseconds align with common APM conventions and human intuition for service response times.

---

## Metadata Section

### `metadata`

| Property | Value |
| -------- | ----- |
| Type | `object` (string key-value pairs) |
| Required | No |
| Example | `{ "userId": "usr_123", "requestId": "req_abc" }` |

An open key-value map for additional, event-specific attributes not covered by the typed fields above.

`metadata` is intended for **extensibility**. Producers may attach domain-specific context (request IDs, feature flags, error codes) without requiring a contract revision for every new attribute.

Guidelines:

- Prefer typed contract fields for attributes that are universal across the platform.
- Use `metadata` for service-specific or experimental attributes.
- Keys should be `camelCase` strings; values should be strings, numbers, or booleans.
- Avoid high-cardinality or sensitive values (passwords, tokens, PII) unless explicitly approved by data governance policy.

## Versioning Strategy

### Why `schemaVersion` exists

Obsyl will evolve. New event types, fields, and semantics will be introduced as the platform matures. `schemaVersion` gives every consumer an unambiguous signal about which contract rules apply to a given event.

Without explicit versioning, producers and consumers silently diverge — leading to dropped events, mis-parsed fields, and incidents that are difficult to diagnose.

### Backward compatibility

When introducing a new schema version, follow these principles:

1. **Additive changes within a version are preferred.** New optional fields may be added to v1 without incrementing the version, provided all existing consumers can ignore unknown fields.
2. **Breaking changes require a new version.** Removing fields, changing types, or altering required-field semantics must result in a new schema version (e.g. v2).
3. **Consumers must support all active versions.** During a migration window, ingestion and processing pipelines should accept both the current and previous schema versions.
4. **Producers should emit the lowest version that satisfies their needs.** Do not emit v2 fields while declaring `schemaVersion: "1"`.

### Future versions

| Version | Document | Notes |
| ------- | -------- | ----- |
| v1 | `telemetry-event-v1.md` | Initial contract — LOG, METRIC, TRACE |
| v2 | `telemetry-event-v2.md` (planned) | TBD — candidate additions include histogram metrics, trace status codes, resource attributes, and baggage propagation |

When v2 is introduced, this document remains the authoritative reference for v1 events. Both documents coexist; `schemaVersion` determines which applies.
